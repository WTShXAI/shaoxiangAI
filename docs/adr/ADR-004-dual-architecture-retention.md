# ADR-004: 双层架构保留决策

**状态**: ✅ 已实施

**日期**: 2026-06-20

**决策者**: 架构团队

---

## 背景

随着系统复杂度提升，后端 API 服务与预测引擎耦合度增加。需要决定是否将后端 API 和预测引擎拆分为独立服务（微服务），或保持当前双层架构。

## 决策

保留当前双层架构：后端 FastAPI 服务 (`backend/`) 作为 API 层，预测引擎 (`predictors/`, `pipeline/`, `rules/`) 作为引擎层。两层通过 Python 模块直接调用，不引入网络通信。

## 选项对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **双层架构（保留）** | 低延迟、无序列化开销、部署简单 | 无法独立扩缩容 |
| 微服务拆分 | 独立部署、独立扩缩容 | 网络延迟、序列化开销、部署复杂度高 |
| 单体架构 | 最简单 | 无法分离关注点 |

## 后果

**正面**:
- 预测延迟低（毫秒级，无网络开销）
- 单进程部署，运维简单
- 关注点分离：API 层处理 HTTP，引擎层处理预测逻辑

**负面**:
- 引擎层故障会影响整个进程
- 无法独立扩缩 API 和预测引擎
- 未来规模扩大时可能需要拆分
