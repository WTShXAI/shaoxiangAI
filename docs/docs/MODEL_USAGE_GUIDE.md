## 1. 概述

### 1.1 系统简介

哨响AI 是一个足球比赛预测 AI 系统，核心功能是：给定一场比赛的特征数据，输出主胜(H)、平局(D)、客胜(A)三个结果的概率。

### 1.2 核心架构

```
用户请求
    │
    ▼
Commander Agent (总指挥官)
    │ 分析任务类型，调度专家
    │
    ├── DataAnalyst Agent (数据分析)
    │   └── 特征提取、数据解读
    │
    ├── MathAgent (数值计算)
    │   ├── 三层降级: LLM → ModelBridge → 规则
    │   └── ModelBridge → EnsembleTrainer (真实模型)
    │       ├── XGBoost (权重 0.69)
    │       ├── Ridge (权重 0.05)
    │       └── Heuristic (权重 0.26)
    │
    └── Explainer Agent (中文解释)
        └── 翻译为用户友好的中文建议
```

### 1.3 关键概念

| 概念 | 说明 |
|------|------|
| **ModelBridge** | Agent 系统访问真实 ML 模型的唯一入口 |
| **EnsembleTrainer** | 集成模型训练器，包含 XGBoost + Ridge + Heuristic |
| **三层降级** | LLM → 真实模型 → 规则兜底，优先使用高可靠路径 |
| **硬编码值** | 40/28/32 是规则降级时的固定起始值，**不是**真实模型预测 |

### 1.4 快速开始

```python
# 方法 1：通过 ModelBridge 直接预测
from agents.model_bridge import get_model_bridge

bridge = get_model_bridge()
if bridge.available:
    result = bridge.predict({
        "a1": 0.15, "a2": 0.62, "a3": 0.48,
        "a4": 0.09, "a5": 0.12, "a6": 0.02,
        "sigma_trap": 0.03, "lambda_crush": 1.12,
        "epsilon_senti": 0.55, "rank_diff_factor": 0.20,
        "form_momentum": 0.10, "h2h_factor": 0.05,
        "rank_factor": 0.50, "form_factor": 0.55,
        "aerial_advantage": 1.05, "press_intensity": 0.12,
        "card_risk": 0.08, "beta_dev": 0.02,
        "delta_fatigue": 0.95,
    })
    print(result)  # {"home": 0.4521, "draw": 0.2833, "away": 0.2646}
else:
    print("模型未加载！")

# 方法 2：通过 Agent 系统预测
from agents.commander import CommanderAgent
from agents.state import create_initial_state

commander = CommanderAgent()
state = create_initial_state(match_data={...})
result = commander.invoke(state)
```

## 2. 输入特征

### 2.1 特征总览

模型需要 19 个核心特征作为输入。缺少的特征会用默认值自动填充，但默认值比例过高会降低预测质量。

### 2.2 特征详细说明

#### 2.2.1 核心因子 A1-A6

##### A1: 盘口价值因子

- **含义**：基于赔率价值缺口和波动率计算的综合因子。正值表示主队盘口有优势，负值表示客队盘口有优势。
- **取值范围**：[-1.0, 1.0]
- **默认值**：0.0
- **计算公式**：A1 = V_value × (1 + σ_trap)
- **解读**：
  - A1 > 0.2 → 主队在盘口层面有明显优势
  - A1 ≈ 0 → 盘口无明显倾向
  - A1 < -0.2 → 客队在盘口层面有明显优势

##### A2: 基本面优势因子

- **含义**：综合战术克制、体能优势、逼抢对抗的基本面因子。
- **取值范围**：[0.0, 1.0]
- **默认值**：0.5
- **计算**：(战术克制 + 体能优势 + 逼抢对抗) / 3
- **解读**：
  - A2 > 0.6 → 主队基本面明显占优
  - A2 ≈ 0.5 → 基本面均衡
  - A2 < 0.4 → 客队基本面占优

##### A3: 市场情绪因子

- **含义**：综合 NLP 情感、新闻冲击、大户信号的情绪因子。
- **取值范围**：[0.0, 1.0]
- **默认值**：0.5
- **计算**：(ε_senti + 新闻冲击 + 大户信号) / 3
- **解读**：
  - A3 > 0.6 → 市场情绪偏向主队
  - A3 ≈ 0.5 → 市场情绪中性
  - A3 < 0.4 → 市场情绪偏向客队

##### A4: 盘口-基本面协同因子

- **含义**：A1 和 A2 的乘积。当盘口价值和基本面优势同向时信号放大，背离时削弱。
- **取值范围**：[-0.5, 0.5]
- **默认值**：0.0
- **公式**：A4 = A1 × A2
- **解读**：
  - A4 > 0.1 → 盘口与基本面共振，主队强信号
  - A4 ≈ 0 → 盘口与基本面背离或其中一方中性
  - A4 < -0.1 → 盘口与基本面共振指向客队

##### A5: 波动调整信号因子

- **含义**：用波动率对 A1 盘口信号做风险调整。高波动时衰减，低波动时接近 A1。
- **取值范围**：[-0.5, 0.5]
- **默认值**：0.0
- **公式**：A5 = A1 / (1 + |σ_trap|)
- **解读**：
  - A5 与 A1 同号但幅度更保守
  - 波动率越大，A5 相对 A1 的衰减越多

##### A6: 市场分歧度因子

- **含义**：市场情绪偏离中性的程度，乘以盘口方向。
- **取值范围**：[-0.5, 0.5]
- **默认值**：0.0
- **公式**：A6 = |A3 - 0.5| × sign(A1)
- **解读**：
  - A6 > 0 → 市场一致看好主队方向
  - A6 ≈ 0 → 市场分歧或盘口方向不明确
  - A6 < 0 → 市场一致看好客队方向

---

#### 2.2.2 盘口动态特征

| 特征 | 中文 | 范围 | 默认值 | 说明 |
|------|------|------|--------|------|
| `sigma_trap` | 异常波动率 | [0, ∞) | 0.0 | 赔率对数收益率的归一化均值 |
| `lambda_crush` | 战术克制系数 | [0, ∞) | 1.0 | 加权实力比 × (1 - e^(-压制率)) |
| `epsilon_senti` | 情绪偏差因子 | [0, 1] | 0.5 | NLP情感 × Sigmoid修正 |
| `beta_dev` | 盘口偏差指数 | [0, ∞) | 0.0 | |理论让球 - 实际让球| × 100% |

**sigma_trap 详解**：

- 含义：赔率的异常波动程度，反映市场是否存在"陷阱"
- 公式：σ_trap = mean(|ln(赔率_t1 / 赔率_t0)|) × (日均交易量/24h)
- 典型值：
  - 0.0-0.05：市场稳定
  - 0.05-0.15：轻微波动
  - \> 0.15：异常波动，需警惕"赔率陷阱"

**lambda_crush 详解**：

- 含义：一方战术上对另一方的克制程度
- 典型值：
  - 0.8-1.2：战术均势
  - \> 1.5：一方有明显战术克制
  - < 0.5：战术上处于明显劣势

---

#### 2.2.3 赛事基本面特征

| 特征 | 中文 | 范围 | 默认值 | 说明 |
|------|------|------|--------|------|
| `rank_diff_factor` | 排名差距因子 | [-1, 1] | 0.0 | 两队排名差异归一化 |
| `form_momentum` | 状态动能 | [-1, 1] | 0.0 | 近期表现趋势动量 |
| `h2h_factor` | 交锋历史因子 | [-1, 1] | 0.0 | 历史交锋倾向 |
| `rank_factor` | 排名因子 | [0, 1] | 0.5 | 综合排名信号 |
| `form_factor` | 状态因子 | [0, 1] | 0.5 | 近期状态评估 |
| `aerial_advantage` | 空中优势 | [0, ∞) | 1.0 | 攻方争顶/守方防空 |
| `press_intensity` | 逼抢强度 | [0, ∞) | 0.0 | 逼抢次数/传球成功率 |
| `card_risk` | 红黄牌风险 | [0, 1] | 0.0 | 裁判严格度修正 |
| `delta_fatigue` | 体能衰减 | [0, 1] | 1.0 | e^(-0.05×周中强度) |

**delta_fatigue 详解**：

- 含义：主队体能状态，周中比赛越多值越低
- 典型值：
  - 1.0：无周中比赛，体能充沛
  - 0.85-0.95：有一场周中比赛
  - < 0.80：多场周中比赛，体能堪忧

## 3. 输出含义

### 3.1 输出格式

```json
{
    "home": 0.4521,
    "draw": 0.2833,
    "away": 0.2646
}
```

| 字段 | 含义 | 标签 |
|------|------|------|
| `home` | 主队胜概率 | H |
| `draw` | 平局概率 | D |
| `away` | 客队胜概率 | A |

**核心规则：home + draw + away = 1.0**（经归一化保证，四舍五入误差 ±0.0001）

### 3.2 概率的真实含义

模型输出的概率 **不是** "这场比赛必定以 XX% 的概率主胜"，而是：

> "在历史数据中，具有相似特征的比赛中，约 XX% 最终由主队获胜。"

因此：

- `home: 0.60` 意味着：历史上类似特征的比赛中，约 60% 主队获胜
- `home: 0.35` 意味着：主胜、平局、客胜的可能性差不多

### 3.3 与赔率隐含概率的关系

| 对比 | 模型概率 > 赔率隐含概率 | 模型概率 < 赔率隐含概率 |
|------|--------------------------|--------------------------|
| 含义 | 模型认为该结果被市场低估 | 模型认为该结果被市场高估 |
| 决策 | 可能存在价值投注机会 | 不建议押注该结果 |
| 术语 | Positive Edge | Negative Edge |

**示例**：

```
模型预测: home=0.45, draw=0.28, away=0.27
赔率: home=2.50, draw=3.20, away=2.80

赔率隐含概率: home=0.40, draw=0.3125, away=0.3571
(计算方式: 1/赔率)

价值分析:
- home: 0.45 - 0.40 = +0.05 (5% 优势) ← 价值投注
- draw: 0.28 - 0.3125 = -0.0325
- away: 0.27 - 0.3571 = -0.0871
```

### 3.4 ⚠️ 硬编码值警告

如果输出始终为 `home=0.40, draw=0.28, away=0.32`，说明**模型未加载**，系统正在使用规则降级的硬编码基础值。

**识别方法**：
1. 三次以上预测结果完全相同
2. 审计日志中 `_source` 字段为 `rules_fallback`
3. `home` 约等于 0.40 且 `draw` 约等于 0.28 且 `away` 约等于 0.32

**此概率不可信，请勿用于任何决策！**

## 4. 置信度解释

### 4.1 置信度分级

| 等级 | 条件 | 图标 | 含义 | 行动建议 |
|------|------|------|------|----------|
| **高置信** | 最高概率 > 0.50 | 🟢 | 模型对某一结果有明确倾向 | 可作为分析参考 |
| **中置信** | 最高概率 0.40-0.50 | 🟡 | 模型有一定倾向但不强烈 | 谨慎参考，结合其他信息 |
| **低置信** | 最高概率 < 0.40 | 🔴 | 比赛结果高度不确定 | 建议观望，不建议投注 |

### 4.2 置信度计算

系统使用以下公式计算置信度：

```python
# 当使用真实模型时
confidence = min(0.85, 0.45 + max_prob * 0.4 + feature_count * 0.015)

# 当使用规则降级时（不可靠）
confidence = min(0.85, 0.35 + max_prob * 0.4 + feature_count * 0.015)
```

**影响因子**：

- **最高概率**：越高 → 置信度越高
- **特征数量**：越多 → 信息越充分 → 置信度越高
- **数据来源**：真实模型起始 0.45，规则降级起始 0.35

### 4.3 置信度解读示例

**高置信案例**：
```
概率: home=0.62, draw=0.20, away=0.18
最高概率: 0.62 (> 0.50)
置信度: ~0.70
→ 模型明确倾向主胜，但仍有 38% 的概率出现平局或客胜
```

**中置信案例**：
```
概率: home=0.45, draw=0.30, away=0.25
最高概率: 0.45 (0.40-0.50)
置信度: ~0.63
→ 模型倾向主胜，但不确定性较高
```

**低置信案例**：
```
概率: home=0.35, draw=0.33, away=0.32
最高概率: 0.35 (< 0.40)
置信度: ~0.59
→ 三种结果概率接近，模型无法有效区分
```

### 4.4 风险等级

| 风险等级 | 条件 | 对应置信度 |
|----------|------|-----------|
| LOW | 最高概率 ≥ 0.50 | 高置信 |
| MEDIUM | 最高概率 0.38-0.50 | 中置信 |
| HIGH | 最高概率 < 0.38 | 低置信 |

## 5. 版本管理

### 5.1 版本号规则

| 版本位 | 变更条件 | 示例 |
|--------|----------|------|
| **主版本号 (Major)** | 模型架构变更 | 2.0 → 3.0（新增集成模型） |
| **次版本号 (Minor)** | 超参数/特征/数据更新 | 3.0 → 3.1（Optuna 权重优化） |
| **修订号 (Patch)** | Bug 修复/配置微调 | 3.1 → 3.1.1（修复默认值） |

### 5.2 模型文件命名

| 文件名 | 含义 | 优先级 |
|--------|------|--------|
| `football_balanced_production.joblib` | 生产稳定版 | 🥇 最高 |
| `footballai_v4_latest.joblib` | V4 多 seed 版 | 🥈 次高 |
| `footballai_expert_latest.joblib` | 专家版 | 🥉 第三 |
| `footballai_v4_ensemble_*.joblib` | V4 集成（按时间戳） | 按时间降序 |
| `auto_balanced_*.joblib` | 自动训练版 | 按时间降序 |

> ModelBridge 自动按优先级查找模型文件。

### 5.3 模型升级流程

```
1. 准备候选模型
   ├── 在新数据上训练
   ├── 运行 Walk-Forward 回测
   └── 确认评估指标不低于当前版本

2. 验证候选模型
   ├── 检查特征数与配置匹配
   ├── 检查概率分布是否合理
   └── 检查文件大小 > 1MB

3. 部署候选模型
   ├── 将候选文件重命名为 football_balanced_production.joblib
   ├── 重启 ModelBridge（或重启服务）
   └── 运行测试预测验证

4. 更新文档
   ├── 更新 MODEL_CARD.md
   ├── 更新本使用说明书的版本信息
   └── 通知下游使用者

5. 监控
   ├── 观察前 24 小时的预测日志
   ├── 检查是否有硬编码值出现
   └── 对比新旧模型的预测差异
```

### 5.4 回滚流程

如果新模型出现问题：

```bash
# 1. 恢复旧模型文件
cp saved_models/football_balanced_production.joblib.bak saved_models/football_balanced_production.joblib

# 2. 重启服务
# 根据部署方式重启

# 3. 验证回滚
python -c "
from agents.model_bridge import get_model_bridge
bridge = get_model_bridge()
print(f'Available: {bridge.available}')
result = bridge.predict({'a1': 0.1, 'a2': 0.6, 'a3': 0.5})
print(f'Test prediction: {result}')
"
```

## 6. 故障排查

### 6.1 常见错误与解决方法

#### 错误 1：预测结果始终是 H=0.40/D=0.28/A=0.32

| 项目 | 说明 |
|------|------|
| **症状** | 所有预测结果几乎相同 |
| **根因** | ModelBridge 未加载真实模型，系统使用规则降级的硬编码值 |
| **检查** | `ls -la saved_models/football_balanced_production.joblib` |
| **解决** | 确认模型文件存在且大小 > 1MB，重启服务 |

#### 错误 2：ModelBridge 初始化失败

| 项目 | 说明 |
|------|------|
| **症状** | 日志中出现 `[ModelBridge] 模型加载失败` |
| **根因** | 模型文件缺失、损坏、或格式不兼容 |
| **检查** | 查看日志 `grep "\[ModelBridge\]" logs/*.log` |
| **解决** | 1. 确认文件存在 2. 确认文件大小合理 3. 尝试手动加载 `joblib.load()` |

#### 错误 3：特征数量不匹配

| 项目 | 说明 |
|------|------|
| **症状** | 预测时出现特征维度错误 |
| **根因** | 模型训练时的特征列表与当前 config.yaml 不一致 |
| **检查** | 对比模型的 `feature_names` 与 config.yaml |
| **解决** | 使用模型自带的 feature_names，不要依赖 config.yaml |

#### 错误 4：概率之和不为 1

| 项目 | 说明 |
|------|------|
| **症状** | home + draw + away ≠ 1.0 |
| **根因** | V4 格式多 seed 平均后归一化失败 |
| **检查** | 在预测后手动验证概率和 |
| **解决** | 手动归一化：`p / sum(p)` |

#### 错误 5：Ollama 连接失败

| 项目 | 说明 |
|------|------|
| **症状** | 日志中 `Ollama 连接失败` |
| **根因** | Ollama 服务未启动或模型未下载 |
| **影响** | 不影响 ModelBridge 路径，仅影响 LLM 路径 |
| **解决** | 启动 Ollama：`ollama serve`，然后拉取模型 |

#### 错误 6：内存不足

| 项目 | 说明 |
|------|------|
| **症状** | 加载模型时 OOM |
| **根因** | 模型文件约 12MB，加载时内存峰值约 200MB |
| **检查** | `free -h` 查看内存 |
| **解决** | 确保有至少 512MB 可用内存 |

## 7. 复盘流程

> 面向非技术人员的 5 步复盘指南，无需编写代码。

### 第 1 步：查看预测记录 📋

**操作**：打开 `logs/predictions/` 文件夹，找到对应日期的文件

**查找方法**：
- 按日期找到文件（如 `2026-06-01.json`）
- 用文本搜索队伍名或 match_id

**关注字段**：
- `probabilities`：模型给出的 H/D/A 概率
- `confidence`：置信度分数
- `risk_level`：风险等级

### 第 2 步：确认概率来源 ✅

**操作**：查看记录中的 `_source` 字段

| _source 值 | 含义 | 可信度 |
|------------|------|--------|
| `EnsembleTrainer` | 来自真实 ML 模型 | ✅ 可信 |
| `rules_fallback` | 来自硬编码规则 | ❌ 不可信 |
| `ollama_llm` | 来自 LLM 推理 | ⚠️ 需验证 |

**同时检查** `_feature_default_ratio`：

| 比值 | 含义 |
|------|------|
| < 0.20 | 输入数据充分，预测可参考 |
| 0.20-0.50 | 部分数据缺失，精度受影响 |
| > 0.50 | 数据严重不足，预测不可靠 |

### 第 3 步：比对实际结果 ⚽

**操作**：获取比赛实际结果，与预测对比

| 预测结果 | 实际结果 | 判断 |
|----------|----------|------|
| home=0.60 (最高) | 主胜 H | ✅ 预测正确 |
| home=0.60 (最高) | 平局 D | ❌ 预测错误 |
| home=0.35 (最高) | 客胜 A | ❌ 预测错误，且低置信 |

### 第 4 步：分析偏差原因 🔍

如果预测错误，从以下维度分析：

| 维度 | 问题 | 如何判断 |
|------|------|----------|
| **数据质量** | 输入特征是否充分？ | `_feature_default_ratio` > 0.50? |
| **置信度** | 模型是否真的"有把握"？ | `confidence` < 0.50? |
| **概率分布** | 是否三线接近？ | 最高概率 < 0.40? |
| **联赛特性** | 该联赛是否是弱项？ | 参考 MODEL_CARD 分联赛准确率 |
| **硬编码** | 是否用了假概率？ | `_source` = `rules_fallback`? |

### 第 5 步：记录与改进 📝

**操作**：将复盘结论记录下来

**记录模板**：

```
比赛：Arsenal vs Chelsea | 联赛：Premier League
预测：H=0.45 D=0.28 A=0.27 (中置信)
实际：平局 D
结果：❌ 预测错误
偏差分析：模型倾向主胜但实际平局，英超平局率约 24%，
         该联赛模型准确率仅 41.1%，属于弱项联赛
改进建议：对英超平局场景增加额外分析
```

**改进路径**：

- 发现数据质量问题 → 反馈给数据采集团队
- 发现硬编码问题 → 按 [故障排查](#6-故障排查) 修复
- 发现某联赛持续不准 → 标记为弱项，单独评估
- 发现新特征可提升预测 → 提交给特征工程团队
